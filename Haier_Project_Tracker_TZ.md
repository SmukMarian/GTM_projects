# Техническое задание  
на разработку локального веб‑приложения «Projects Tracker»

---

## 1. Назначение и цель системы

Система предназначена для **личного ведения проектов по запуску новых продуктов на рынок**, с фокусом на:

- управлении продуктовыми группами и проектами;
- ведении go-to-market этапов;
- управлении задачами и подзадачами;
- хранении характеристик продукта **на двух языках (RU/EN) рядом**;
- хранении медиа (обложки и галереи);
- гибкой кастомизации структуры полей и этапов;
- импорте/экспорте данных через Excel.

Система используется **одним человеком** (заказчиком ТЗ), многопользовательский режим и сложные роли не требуются.

---

## 2. Общие требования

### 2.1. Локальная работа

- Приложение работает полностью **локально** на одном компьютере.
- Доступ к интерфейсу осуществляется через **ссылку в браузере** (локальный адрес).
- Запуск производится через **иконку / ярлык** на рабочем столе или в меню:
  - при запуске поднимается локальный сервис;
  - автоматически открывается основная страница приложения в браузере.

### 2.2. Отсутствие зависимости от интернета

- Для основной работы (просмотр, редактирование, импорт/экспорт, архивирование) интернет не требуется.

### 2.3. Язык интерфейса и данных

- Язык интерфейса — русский.
- Данные характеристик продукта хранятся и отображаются **на двух языках (RU и EN) одновременно**, RU и EN значения находятся рядом.

### 2.4. Состояние проекта

- Не требуется «облегчённый» MVP.
- **Первая версия** должна реализовывать **весь описанный функционал**.

### 2.5. Работа правой кнопкой мыши и операции удаления

- Во всех списках и карточках сущностей (продуктовые группы, проекты, этапы, задачи, подзадачи, файлы, изображения, секции и поля характеристик) должна поддерживаться работа **правой кнопкой мыши**:
  - по правому клику открывается контекстное меню с основными действиями (просмотр/открыть, редактировать, дублировать, архивировать, удалить — где применимо).
  - на устройствах без правой кнопки мыши (трекпад) должна быть предусмотрена эквивалентная комбинация (например, долгий тап или `Ctrl + click`).
- Для **всех сущностей**, где это логично, должна быть возможность **полного удаления**:
  - удаление должно сопровождаться подтверждением (диалог «Вы уверены?»);
  - при удалении родительской сущности должны корректно удаляться/отвязаться связанные дочерние элементы (например, при удалении проекта удаляются его задачи, этапы, характеристики, файлы, изображения).

## 3. Структура системы и сущности

### 3.1. Основные сущности

1. **Продуктовая группа**  
   Логическое объединение проектов (например, «Robot VC», «Air Treatment», «Kettles»).

2. **Проект**  
   Конкретный проект запуска/развития продукта или линейки.

3. **Этап go-to-market (GTM‑этап)**  
   Шаги процесса вывода продукта на рынок (кастомизируемые, с возможностью сохранения в шаблоны).

4. **Задача**  
   Единица работы внутри проекта, может иметь подзадачи и комментарии.

5. **Подзадача**  
   Детализация задачи (чек‑лист внутри задачи).

6. **Характеристика продукта**  
   Набор параметров продукта, разделённых на секции и поля, **для RU и EN одновременно**.

7. **Шаблон структуры**  
   - Шаблон GTM‑этапов.
   - Шаблон структуры характеристик.

8. **Файл**  
   Прикреплённые документы к проекту.

9. **Изображение**  
   Фотографии/рендеры для галереи проекта и обложки.

10. **Комментарий**  
    Комментарии к проектам и задачам.

## 4. Главная страница — Дашборд

### 4.1. Назначение

Дать быстрый обзор ситуации по всем продуктовым группам и проектам:

- сколько активных проектов;
- в каких статусах;
- какие ближайшие важные даты;
- какие проекты/группы требуют внимания.

### 4.2. Элементы дашборда

1. **Сводка по статусам проектов**

- Количество проектов в статусах:
  - «Активный»;
  - «Закрыт»;
  - «Архивирован» (по умолчанию не считается в активные).
- Отображение в виде карточек/цифр/диаграмм.

2. **Сводка по продуктовым группам**

Для каждой продуктовой группы отображается:

- название группы;
- количество активных проектов;
- наличие проектов с просроченными этапами/задачами (индикатор риска);
- возможность кликнуть и перейти на страницу группы.

3. **Ближайшие важные даты**

Список:

- ближайшие дедлайны GTM‑этапов;
- ближайшие сроки по задачам (отмеченным как важные/критичные).

Таблица:

- Проект;
- Продуктовая группа;
- Этап / задача;
- Плановая дата;
- Дней до / с просрочки (если есть).

4. **Последние изменения**

Лента:

- последние изменения статусов проектов;
- изменение GTM‑этапов;
- закрытие крупных задач.

### 4.3. Функционал дашборда

- Переход в продуктовую группу по клику на группу.
- Переход в карточку проекта по клику по строке события/дедлайна.
- Фильтры:
  - по продуктовой группе;
  - по бренду;
  - по статусу проекта (активные / закрытые / архивные).

- Контекстное меню по правому клику:
  - для карточек групп и проектов — быстрые действия (открыть, архивировать, удалить).

## 5. Продуктовые группы

### 5.1. Назначение

Рабочее пространство для конкретной категории продуктов.

### 5.2. Структура данных Продуктовой группы

Поля:

- Название группы;
- Описание (опционально);
- Статус:
  - Активна;
  - Архивирована;
- Бренды, с которыми связана (опционально, может быть несколько);
- Дополнительные пользовательские поля (создаются вручную и могут использоваться в фильтрах).

### 5.3. Список проектов в группе

Отображение:

- в виде карточек (плитка);
- и в виде таблицы (переключаемые режимы).

**Карточка проекта в группе** показывает:

- квадратную обложку (из галереи проекта);
- название проекта;
- бренд;
- статус проекта;
- текущий GTM‑этап;
- прогресс (визуальная шкала по GTM‑этапам);
- индикатор риска (если есть просроченные этапы/задачи).

### 5.4. Фильтры в продуктовой группе

Обязательные фильтры:

- по бренду;
- по статусу проекта (Активные / Закрытые / Архивные);
- по плановому периоду запуска (например, год/квартал);
- по текущему GTM‑этапу.

Дополнительно:

- сохранённые пользовательские фильтры;
- фильтрация по кастомным полям (если они созданы для группы/проекта).

### 5.5. Операции с продуктовой группой

- Создание новой группы;
- Редактирование группы;
- Архивирование группы:
  - группа помечается как «архивная»;
  - по умолчанию не отображается в обычном списке;
  - при включении фильтра «Показывать архив» — видна;
- Восстановление группы из архива;
- **Удаление группы**:
  - доступно из панели действий и контекстного меню (правая кнопка мыши);
  - при удалении должно быть подтверждение;
  - при удалении группы необходимо определить поведение связанных проектов (либо запрещать удаление при наличии проектов, либо принудительно удалять/переносить проекты — вариант согласуется на этапе реализации).

Контекстное меню по правому клику на группе:

- открыть;
- редактировать;
- архивировать / восстановить;
- удалить.

## 6. Проекты

### 6.1. Структура проекта

Поля проекта:

- Название проекта;
- Продуктовая группа (ссылка);
- Бренд;
- Рынок/регион (текст);
- Краткое описание проекта;
- Расширенное описание проекта;
- Статус проекта:
  - Активный;
  - Закрыт;
  - Архивирован;
- Текущий GTM‑этап (выбирается из списка этапов проекта);
- Плановая дата запуска;
- Фактическая дата запуска (при наличии);
- Важность/приоритет (низкий/средний/высокий — опционально);
- Пользовательские поля (настраиваемые).

### 6.2. Основной экран проекта

Верхняя часть:

- Квадратная обложка проекта (из галереи);
- Название проекта;
- Бренд, группа, рынок;
- Статус и текущий GTM‑этап;
- Плановая и фактическая дата запуска;
- Краткое описание.

Ниже — вкладки:

1. Go-to-market этапы;
2. Характеристики продукта (RU/EN);
3. Задачи и подзадачи;
4. Галерея;
5. Файлы;
6. Комментарии;
7. История изменений.

На уровне проекта:

- Возможность **редактировать** данные проекта;
- Возможность **закрыть проект** (статус «Закрыт»);
- Возможность **архивировать проект**;
- Возможность **удалить проект** (с подтверждением), включая его этапы, задачи, подзадачи, характеристики, файлы и изображения.

Контекстное меню по правому клику (в списке проектов и в карточке):

- открыть;
- редактировать;
- закрыть;
- архивировать / восстановить;
- удалить;
- при необходимости — дублировать (создание нового проекта на основе структуры текущего).

## 7. Go-to-market этапы (GTM‑этапы)

### 7.1. Общая логика

- Этапы полностью **кастомизируемые** для каждого проекта.
- Пользователь может:
  - добавлять этапы через кнопку «плюс»;
  - редактировать названия и описания;
  - менять порядок этапов;
  - задавать плановые и фактические даты;
  - отмечать статусы.
- Сохранение наборов этапов как **шаблонов**:
  - шаблоны можно применять к другим проектам;
  - шаблоны можно выгружать и загружать через Excel.

### 7.2. Структура GTM‑этапа

Поля:

- Название этапа;
- Описание (что считается завершением);
- Порядковый номер (порядок в цепочке);
- Плановая дата начала;
- Плановая дата окончания;
- Фактическая дата завершения;
- Статус:
  - Не начат;
  - В работе;
  - Завершён;
  - Отменён;
- Признак риска (ручная отметка или автоматическая по просрочке);
- Чек‑лист подэлементов (список текстовых пунктов с галочками).

### 7.3. Управление этапами

- Добавление этапа (кнопка «+»);
- Редактирование этапа;
- Изменение порядка этапов;
- Удаление этапа (с подтверждением);
- Создание **шаблона этапов** по текущему проекту;
- Применение шаблона к новым или существующим проектам;
- Экспорт структуры GTM‑этапов в Excel;
- Импорт GTM‑структуры из Excel.

Контекстное меню по правому клику на этапе:

- редактировать;
- добавить чек‑пункты/подэтапы;
- дублировать этап;
- удалить этап.

## 8. Характеристики продукта (две языковые колонки)

### 8.1. Структура характеристик

Характеристики организованы в виде:

- **Секций** (группы полей);
- **Полей** внутри секций.

Примеры секций:

- Основные параметры;
- Функции;
- Комплектация;
- УТП и позиционирование;
- Конкуренты (опционально).

Каждое поле характеристики содержит:

- Название поля **на русском** (Label RU);
- Название поля **на английском** (Label EN);
- Значение **на русском** (Value RU);
- Значение **на английском** (Value EN);
- Тип поля:
  - текстовое поле;
  - число;
  - список;
  - флажок (да/нет);
  - при необходимости — другие типы.

На экране:

- RU и EN значения отображаются **рядом**, чтобы удобно было одновременно видеть и редактировать оба языка.

### 8.2. Кастомизация структуры характеристик

Возможности пользователя:

- добавлять новые секции;
- переименовывать и удалять секции;
- добавлять новые поля в секции;
- задавать Label RU и Label EN для каждого поля;
- выбирать тип поля;
- менять порядок полей и секций.

Для секций и полей:

- доступны операции:
  - редактирование;
  - удаление;
  - при необходимости — дублирование поля/секции.

Все эти действия должны быть доступны как через обычные элементы управления, так и через **контекстное меню правой кнопки мыши**.

### 8.3. Шаблоны характеристик

- Возможность создать **шаблон характеристик**:
  - например, «Шаблон робота», «Шаблон утюга» и т.п.
- Шаблон включает:
  - набор секций;
  - список полей в каждой секции;
  - типы полей;
  - Label RU/EN (без значений Value RU/EN).
- При создании нового проекта:
  - можно выбрать шаблон, и структура характеристик подтягивается.
- Возможность **скопировать структуру** характеристик из другого проекта:
  - копируются только секции, поля, типы и Label RU/EN;
  - значения не копируются (по умолчанию).

### 8.4. Импорт/экспорт характеристик через Excel

**Экспорт:**

- Выгрузка характеристик проекта в Excel с колонками:
  - Секция;
  - Поле (Label RU);
  - Поле (Label EN);
  - Значение RU;
  - Значение EN;
  - Тип поля (опционально).

**Импорт:**

- Пользователь скачивает **шаблон Excel** для проекта:
  - в шаблоне уже указаны секции и поля.
- После заполнения RU/EN значений:
  - файл загружается обратно;
  - система сопоставляет записи по секции и названию полей;
  - заполняет соответствующие Value RU/EN.

При ошибках импорта:

- выводится понятное сообщение, какие строки/поля не удалось сопоставить.

## 9. Задачи и подзадачи

### 9.1. Структура задачи

Поля задачи:

- Заголовок;
- Описание задачи (подробный текст);
- Статус:
  - ToDo;
  - В работе;
  - Готово;
- Срок выполнения (дата);
- Признак важности (опционально);
- Привязка к GTM‑этапу (опционально);
- Возможность отметить задачу как закрытую (статус «Готово»).

Дополнительно:

- Каждая задача должна иметь:
  - комментарии;
  - подзадачи (чек‑лист).

### 9.2. Подзадачи (чек‑лист)

- Список элементов внутри задачи:
  - текст подзадачи;
  - флаг «выполнено / не выполнено».
- Подзадачи относятся к одной задаче.

### 9.3. Комментарии к задачам

Для каждой задачи:

- блок комментариев;
- каждый комментарий:
  - текст;
  - дата/время.

Комментарии должны быть доступны для создания, редактирования (опционально) и удаления.

### 9.4. Представления задач

- Список задач проекта (табличный вид);
- Представление «доска»:
  - колонки по статусам (ToDo / В работе / Готово);
- Фильтры:
  - по статусу;
  - по сроку (ближайшие, просроченные);
  - по привязанному GTM‑этапу.

### 9.5. Операции с задачами и подзадачами

Для задач:

- создание;
- редактирование;
- перевод по статусам;
- закрытие (статус «Готово»);
- **удаление задачи** (с подтверждением);
- при необходимости — дублирование (создание копии задачи).

Для подзадач:

- создание;
- редактирование текста;
- отметить выполненной / невыполненной;
- **удаление подзадачи**.

Все эти операции должны быть доступны:

- через элементы управления у каждой задачи/подзадачи;
- через **контекстное меню правой кнопки мыши** по задаче/подзадаче (редактировать, завершить, удалить, дублировать — где применимо).

## 10. Галерея и обложка

### 10.1. Галерея проекта

Функционал:

- Загрузка нескольких изображений;
- Хранение:
  - имя файла;
  - комментарий/подпись;
  - дата добавления;
- Изменение порядка изображений.

Операции:

- добавление изображения;
- редактирование подписи;
- **удаление изображения** (с подтверждением).

Контекстное меню по правому клику на изображении:

- открыть/просмотреть;
- назначить обложкой;
- переименовать/изменить подпись;
- удалить.

### 10.2. Обложка проекта

- Выбор одного изображения из галереи как **обложки**;
- Обложка должна быть **квадратной** (используется соответствующая область или заранее подготовленный квадратный файл);
- Обложка отображается:
  - на карточках проекта в списках;
  - на дашборде;
  - в шапке проекта.

## 11. Файлы и документы

Для каждого проекта:

- Список файлов:
  - название файла;
  - описание;
  - категория (например, «техдок», «маркетинг», «юридические» — список категорий настраиваемый);
  - дата загрузки.

Операции:

- загрузка нового файла;
- редактирование описания и категории;
- скачивание файла;
- **удаление файла** (с подтверждением).

Контекстное меню по правому клику:

- скачать;
- переименовать/изменить описание;
- изменить категорию;
- удалить.

## 12. Комментарии и история

### 12.1. Комментарии к проекту

- Отдельный блок комментариев в карточке проекта;
- Каждый комментарий:
  - текст;
  - дата/время.

Операции:

- добавить комментарий;
- при необходимости — редактировать комментарий;
- **удалять комментарий** (с подтверждением).

### 12.2. История изменений

- Фиксация ключевых изменений:
  - изменение статуса проекта;
  - изменение GTM‑этапов;
  - создание/удаление задач;
  - изменение характеристик (укрупнённо);
  - загрузка/удаление файлов;
  - установка/смена обложки.

История отображается в виде ленты событий внутри проекта.

## 13. Архивирование, скрытие и удаление

### 13.1. Проекты

Статусы:

- Активный;
- Закрыт;
- Архивирован.

Операции:

- перевести проект в статус «Закрыт»;
- архивировать проект;
- восстановить проект из архива;
- **удалить проект** (с подтверждением), включая:
  - задачи и подзадачи;
  - GTM‑этапы;
  - характеристики;
  - файлы;
  - изображения;
  - комментарии.

Фильтры:

- по умолчанию отображаются активные и/или закрытые проекты;
- архивные — только при включении опции «Показывать архив».

### 13.2. Продуктовые группы

Статусы:

- Активна;
- Архивирована.

Операции:

- архивировать;
- восстановить;
- **удалить группу** (с подтверждением и понятной логикой по связанным проектам).

### 13.3. Задачи

- Статус «Готово» означает закрытие задачи;
- Дополнительно:
  - возможность скрывать выполненные задачи через фильтр;
  - возможность **удалить** задачу полностью.

## 14. Импорт/экспорт через Excel (общее описание)

Система должна поддерживать:

1. Импорт и экспорт характеристик проекта (RU/EN);
2. Импорт и экспорт GTM‑этапов и шаблонов этапов;
3. Экспорт списка проектов и их ключевых данных.

Требования:

- Заголовки столбцов должны быть понятными и однозначными;
- Сопоставление при импорте:
  - по названию секций;
  - по Label RU/EN для характеристик;
  - по названиям этапов для GTM;
- При ошибках импорта:
  - выводить понятное сообщение, какие строки/поля не удалось сопоставить и почему.

## 15. Нефункциональные требования

- Корректная работа с:
  - десятками продуктовых групп;
  - сотнями проектов;
  - большим количеством задач и файлов.

- Быстрый отклик интерфейса:
  - изменение статусов, фильтрация и навигация не должны ощущаться медленными.

- Хранение данных:
  - в локальном файле/хранилище (например, единая база данных/файл проекта);
  - возможность ручного копирования этого файла для резервного копирования.

- Резервное копирование:
  - функция «Сохранить резервную копию базы» (экспорт в файл);
  - функция «Восстановить из резервной копии» (загрузка файла).

- Интерфейс:
  - интуитивный, без необходимости специальных технических знаний;
  - все критические операции удаления сопровождаются подтверждением.

На основе этого ТЗ должна быть реализована **полноценная рабочая система**, без упрощений, с учётом всех описанных сущностей, операций, контекстных меню и возможностей удаления/архивирования.
